# Post-train детектор отравления данных (`post_train_guard`)

Слой Security Gate, проверяющий **обученную модель** (post-train) на следы отравления
обучающих данных. Категория `MODEL`, вердикт **ALLOW / REVIEW / BLOCK**.
Банковский домен, AI-security. Работает локально на CPU.

---

## 1. Зачем нужны post-train методы

- **Угроза.** Data poisoning — атакующий подмешивает строки в обучающий датасет, чтобы
  внедрить **бэкдор** (скрытый триггер → нужный атакующему ответ) или деградировать модель.
  Цена для банка: риски 152-ФЗ, штрафы **до 3% оборота / 500 млн ₽**, скомпрометированный
  скоринг/антифрод.
- **Почему именно на уровне модели, а не только данных:**
  1. Часть атак **не видна в сырых данных** (clean-label, тонкие триггеры), но оставляет
     след в том, что модель **выучила** — в её активациях/представлениях.
  2. Атака опасна, только если **прижилась в модели**. Post-train проверяет финальный
     артефакт, который реально поедет в прод.
  3. **Defense-in-depth.** Независимая точка контроля рядом с data-level слоем команды
     (PII / качество / prompt-injection). Строгая граница **data ↔ model**.

---

## 2. От каких угроз защищают

| Угроза | Что делает атакующий | Кто ловит |
|---|---|---|
| **Backdoor / триггер** | строки с триггер-токеном → нужный ответ | SPECTRE, Activation Clustering, RPP |
| **Value-trigger / порча признаков** (таблицы) | спец-значение признака + флип метки | SPECTRE, AC (ранжируют отлично) |
| **Label flip** | переворот меток без смены признаков | **model_diff** (с опорой), kNN (резерв) — для representation-методов это **слепая зона** |
| **Homoglyph** (текст) | латиница→кириллица в тексте | слабо у model-level; точный charset-детектор в резерве (data-level) |

**Граница ответственности:** атаки на сам текст **без обученной BERT-модели** этот модуль
не ловит (нужна модель; rule-детекторы текста вынесены в `reserved/` под отдельный
data-level модуль). Термин — «**мультиформатный**» (таблицы + текст), не «мультимодальный».

---

## 3. Как реализованы и как работают

**Архитектура.** Ансамбль независимых методов × {табличный MLP, дообученный BERT}.
Подключение ко всему движку — **один тонкий адаптер** `PostTrainGuardScanner`
(паттерн соседнего `dataset_guard`). Добавить/убрать метод = одна строка в списке `CHECKS`.

**Два режима работы:**

- **Режим 1 — без эталона (ТРИАЖ).** Есть только кандидатные данные + модель. Методы
  **ранжируют** строки по подозрительности и дают **самокалиброванный** вердикт через
  *пермутационный нуль* (перемешиваем метки, пересчитываем статистику → флаг только если
  реальная аномалия выше фона). Приоритет отдаётся Activation Clustering (единственный
  метод с надёжным самодостаточным гейтом); SPECTRE/RPP — ранжирование/корроборация.
  Итог: **ALLOW / REVIEW** (без BLOCK — абсолютного порога без опоры нет).

- **Режим 2 — с опорой (КАЛИБРОВАННЫЙ ВЕРДИКТ).**
  - *Опорная (старая/принятая) модель* → **model_diff**: сравнение предсказаний двух
    моделей по **Total Variation**; кластер сильных расхождений = целевая закладка → BLOCK,
    широкий дрейф → REVIEW.
  - *Чистый сэмпл данных* → **калибровка**: порог из скоров чистых строк (бутстрап-CI);
    доля кандидата сверх порога значимо выше фона → флаг. Это уже **вердикт** (ведёт
    решение), допускает **BLOCK**.

**Методы (4 шт.):**

1. **SPECTRE** (Hayase et al., 2021) — робастный Spectral Signatures: робастное
   «отбеливание» представлений класса (MAD-инициализация + ковариация по инлайерам) +
   **QUE-скор**, усиливающий отравленное подпространство. На self-test (маскированный
   бэкдор) **AUC 1.0 против 0.02** у ванильного Spectral.
2. **Activation Clustering** (Chen et al., 2018) — внутри каждого класса PCA→KMeans(2);
   если класс расщепляется на два разделённых кластера (по silhouette), меньший (≤45%) —
   кандидаты в отравленные.
3. **RPP** (рандомизированное возмущение предсказаний, Lin et al.) — добавляем гауссов шум
   (в признаки / во входные эмбеддинги BERT), меряем L∞-стабильность предсказания.
   Аномально **стабильные** под шумом строки = подозрение на триггер.
4. **model_diff** (режим 2) — per-row TV между старой и новой моделью.

**Инженерия.** numpy/sklearn/torch, CPU. Табличный MLP реконструируется из `state_dict`
(XGBoost нельзя — нет активаций → обучаем представительный MLP под кейс). BERT — через
`transformers` (graceful skip, если пакета нет). **Источник правды — JSON-отчёт**, не консоль.

## 4. Метрики — насколько хорошо работает

Меряем **раздельно** две вещи: **ранжирование** (ROC-AUC по строкам, есть row-level
ground truth) и **вердикт гейта** (ALLOW/REVIEW/BLOCK).

Бенч: **11 чистых + 33 грязных** прогона. Табличные — полные данные, естественная доля
яда; NLP — подвыборка, документированные атаки инжектированы как **меньшинство 15%**
(соответствует minority-допущению методов; полные 47k на CPU неподъёмны).

### Сильная сторона — РАНЖИРОВАНИЕ
Когда яд — подмножество строк, методы выделяют его в топ почти идеально:

| Датасет / атака | тип | AC | SPECTRE | RPP |
|---|---|---|---|---|
| uci_banknote d2 | таблица, порча признаков | **1.00** | 0.99 | 0.79 |
| creditcard d3 | таблица, порча признаков | 0.99 | 0.98 | 0.98 |
| uci_banknote d3 | таблица, порча признаков | 0.96 | 0.95 | 0.94 |
| taiwan d3 | таблица, value-trigger | 0.95 | 0.79 | 0.88 |
| bitext d3 | текст/BERT, **homoglyph** | 0.85 | **0.90** | 0.23 |
| bitext d2 | текст/BERT, **backdoor** | **0.76** | 0.59 | 0.49 |

### Вердикт гейта (режим 1) и границы
- **Чистое → ALLOW: 8/11** (таблицы 5/6, NLP 3/5). Все ложные срабатывания — на данных с
  **очень тесной природной структурой классов**: bank_churn (тесный кластер ушедших
  клиентов), bitext и banking77 (десятки тесных интент-категорий). Без эталона такая
  структура статистически неотличима от закладки.
- **Документированные NLP-атаки (15% меньшинство): 2/2 → REVIEW** (backdoor, homoglyph).
- **Label flip → слеп** для всех трёх representation-методов — зона model_diff / kNN.
- **«Rank-vs-gate gap»:** часть атак ранжируется на AUC≈1.0, но самодостаточный гейт без
  эталона не ставит абсолютный порог (пермутационный нуль обманывается сильной природной
  структурой, напр. uci/give_me) → ALLOW. На bitext обратное: гейт ревьюит и чистое, и
  атаку (не различает по вердикту), но **ранжирование** выделяет верные строки.

### Ключевой честный вывод
> Без эталона model-level — **сильный ранжировщик** (триаж top-подозрительных строк), но
> **не надёжный классификатор** «чисто/грязно». Поэтому вердикт без калибровки = **REVIEW**
> (триаж, не авто-блок); абсолютный вердикт, включая **BLOCK**, — только в **режиме 2**
> (опорная модель / чистый сэмпл). Не обещаем «ловит всё».

## 5. Сравнение с аналогами

- **vs ванильный Spectral Signatures** — SPECTRE робастнее на маскированных/адаптивных
  бэкдорах (robust whitening + QUE; на self-test AUC 1.0 vs 0.02).
- **vs одиночный метод** — ансамбль покрывает **разные сигнатуры** (тесный кластер vs
  устойчивость под шумом), а model_diff закрывает label_flip — слепую зону representation-методов.
- **vs data-level / pre-train статистика (соседний слой команды)** — независимая точка
  контроля. Pre-train видит сырые паттерны данных; post-train видит, **что модель выучила**
  (clean-label, тонкие триггеры, невидимые в данных). Общая болезнь обоих — порог без
  эталона; у нас решается калибровкой/диффом (режим 2). Вместе = defense-in-depth.
- **vs robust training (SecureLearn и т.п., в резерве)** — не требует переобучения,
  работает с готовой моделью пост-фактум.
- **Позиционирование (честно):** не «ловит всё» и не «первое в мире». Триаж-слой с сильным
  ранжированием; абсолютный вердикт требует опоры. Это осознанный, проверяемый дизайн.
